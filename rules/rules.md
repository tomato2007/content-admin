# Правила разработки Content Admin

Дата актуализации: 2026-04-21

## Назначение документа

Это практические правила, по которым нужно вносить следующие изменения в проект.

Документ обязателен для:

- новых feature-модулей;
- новых Filament-компонентов;
- workflow/action классов;
- publishing integration;
- изменения схемы БД и сидеров.

## 1. Общие правила

- Использовать `declare(strict_types=1);` во всех новых PHP-файлах.
- Соблюдать PSR-12 и прогонять `vendor/bin/pint`.
- Перед завершением шага прогонять `php artisan test`.
- Не использовать `dd()` и `dump()` в коде приложения.
- Не изменять `vendor/`.

## 2. Правила размещения кода

### Пока допустимо

Существующий код в:

- `app/Models`
- `app/Policies`
- `app/Services`
- `app/Observers`
- `app/Filament`

считается рабочим legacy baseline.

### Для новых изменений

Новые крупные сценарии размещать по feature-first модели.

Если изменение достаточно маленькое и напрямую продолжает уже существующий класс, допускается менять текущий legacy файл, но:

- без размазывания новой бизнес-логики по UI;
- без роста технической связности между Filament и publishing;
- с документированием, если изменение усиливает legacy boundary.

Для новых сценариев уровня use case целевой путь по умолчанию:

- `app/Features/<FeatureName>/Application/Actions`

## 3. Правила для Domain слоя

В domain слой относить:

- enum;
- value objects;
- правила статусов;
- словарь терминов;
- ограничения жизненного цикла сущностей.

Нельзя:

- тянуть Filament зависимости в domain;
- тянуть HTTP/UI-логику в domain;
- смешивать platform-specific runtime код с доменными правилами.

## 4. Правила для Application слоя

В application слой относить:

- actions/use cases;
- orchestration сценариев;
- moderation workflow;
- publish orchestration;
- lock/idempotency coordination.

Каждый нетривиальный сценарий должен иметь один явный вход:

- `ApprovePlannedPostAction`
- `RejectPlannedPostAction`
- `PublishPlannedPostAction`
- `ReschedulePlannedPostAction`

или эквивалентный именованный use case.

Если legacy service ещё нужен для обратной совместимости тестов или старых вызовов, он должен становиться тонкой обёрткой над action-классами, а не местом для новой логики.

Нельзя:

- держать сложный workflow в Filament action closure;
- прятать guard conditions только в UI;
- смешивать application orchestration с формированием таблиц/форм.

## 5. Правила для Infrastructure слоя

Сюда относятся:

- Eloquent persistence;
- DB integration;
- CLI/process adapters;
- publisher drivers;
- внешние runtime зависимости.

Обязательные правила:

- machine-specific пути не хардкодить в коде;
- все внешние пути, DSN и runtime refs выносить в config/env;
- драйвер платформы выбирать по `platform.driver`, а не глобально на всё приложение;
- внешние интеграции должны иметь безопасный локальный fallback (`null`/fake/stub).

## 6. Правила для Filament

Filament используется как presentation layer.

Разрешено:

- описывать формы, таблицы, infolists;
- вызывать actions/use cases;
- скрывать/показывать UI по policy;
- показывать preview и агрегированные read-model поля.

Нельзя:

- реализовывать полный доменный workflow только внутри relation manager;
- изменять несколько доменных сущностей напрямую в UI без отдельного action/service;
- использовать Filament closures как единственное место бизнес-правил.

## 7. Правила для авторизации

- Доступ всегда проверяется policy.
- Скрытая кнопка не заменяет policy.
- Для критичных сценариев policy + application guard должны работать вместе.
- Роль пользователя рассматривается в контексте `PlatformAccount`, а не как глобальная роль.

## 8. Правила для модерации и очереди

- Замена поста не должна перетирать исходную запись.
- Удаление должно иметь подтверждаемый поток.
- Финальные статусы нельзя “тихо” переиспользовать.
- Любой переход статуса должен быть трассируем в audit log.
- Негативные сценарии должны тестироваться отдельно, не только happy path.

## 9. Правила для публикации

- `PublishingService` или его будущий модульный эквивалент отвечает за orchestration.
- Driver отвечает только за платформенную публикацию.
- Каждая попытка публикации должна попадать в `PostingHistory`.
- Idempotency должна быть на уровне orchestration и БД, а не только защиты от двойного клика в UI.
- Dry-run и publish должны использовать общий контракт запроса/результата.

## 10. Правила для миграций и сидеров

- Миграции должны быть идемпотентны в смысле повторяемого развёртывания на пустой БД.
- Сидеры dev-only данных должны запускаться только в `local` окружении.
- Продовые или production-like credentials нельзя документировать как dev bootstrap.
- Любой локальный bootstrap пользователь должен быть воспроизводим через `.env.example` и `README_DEV.md`.

## 11. Правила для документации

При изменении архитектурно значимых частей нужно обновлять:

- `docs/architecture.md`, если меняется структура слоёв или направление миграции;
- `docs/domain.md`, если меняется словарь домена, роли, статусы или workflow;
- `rules/rules.md`, если меняются обязательные инженерные правила.

## 12. Definition of done для каждого значимого шага

Шаг считается завершённым только если:

- код соответствует текущему архитектурному направлению;
- `vendor/bin/pint --test` проходит;
- `php artisan test` проходит;
- при необходимости обновлена документация;
- в плане зафиксирован статус выполнения.
