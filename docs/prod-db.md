Нужно реализовать подключение к PostgreSQL базе anecdots и слой данных для постинга Telegram так, чтобы поведение полностью совпадало с текущим прод-скриптом.

Требования:

1. Подключение к БД

• Использовать PostgreSQL.
• Конфиг только через env:
  • DATABASE_URL (приоритетно), либо
  • PGHOST, PGPORT, PGDATABASE, PGUSER, PGPASSWORD, PGSSLMODE.
• Поддержать SSL-режимы disable и require.
• Реализовать пул соединений (не одиночный коннект), healthcheck и корректное закрытие по SIGTERM/SIGINT.

2. Схема данных (источник постов)

• База: anecdots
• Таблица: public.posts
• Использовать как минимум поля:
  • id (PK)
  • content (text поста)
  • media_url (nullable)
  • published_at (nullable timestamptz)
• Считать запись “не опубликована”, если published_at IS NULL.

3. Логика выборки (как сейчас)

• Выбор поста происходит в момент запуска джобы (не заранее).
• Источник, общий пул подходящих не опубликованных записей из public.posts.
• Исключать:
  • пустой/NULL content,
  • слишком короткий или мусорный текст (вынести в функцию-валидатор),
  • записи с хвостами/шапками источников (чистка контента перед публикацией).
• Для обычного слота: случайный пост из подходящего пула.
• Для медиа-слота: случайный пост из подходящего подмножества с media_url IS NOT NULL; если нет, fallback на текстовый.
• После успешной публикации обязательно помечать запись опубликованной (published_at = now()), чтобы исключить повтор.

4. Защита от гонок

• Если запускаются несколько воркеров, одна и та же запись не должна публиковаться дважды.
• Использовать транзакционный паттерн вида:
  • SELECT ... FOR UPDATE SKIP LOCKED
  • затем update/commit в одной транзакции.
• Идемпотентность: при сетевой ошибке между send и mark предусмотреть защиту от дублей (delivery log / dedupe key).

5. Планировщик

• Запуск каждые 10 минут.
• Тихие часы: 00:00-06:00 UTC (в это время не публиковать).
• За запуск публиковать максимум 1 пост.

6. Совместимость с текущим пайплайном

• Оставить интеграцию с текущим Telegram publisher-слоем (доставка/репорты могут жить отдельно, например в SQLite),
но источник контента и факт “опубликован/не опубликован” должны опираться на public.posts в Postgres.
• Вынести SQL и репозиторий в отдельный модуль (PostsRepository), без хардкода в бизнес-логике.

7. Наблюдаемость

• Структурные логи: start pick, picked post id, publish success/fail, mark published success/fail.
• Метрики (минимум): attempts, published_count, skipped_quiet_hours, no_candidates.
• Явные коды ошибок для DB connect timeout/auth/query.

8. Миграции и проверка

• Добавить миграцию/проверку индексов для ускорения выборки:
  • индекс по published_at,
  • при необходимости частичный индекс WHERE published_at IS NULL.
• Добавить smoke script:
  • проверка подключения,
  • SELECT count(*) доступных к публикации,
  • тестовый pick без mark.

9. Тесты

• Unit: валидатор/очистка текста, quiet hours, media-slot fallback.
• Integration (docker postgres): конкурентный pick двумя воркерами без дублей.
• E2E (mock telegram): успешная публикация => published_at проставлен.

10. Артефакты

• .env.example с полным списком переменных.
• README с командами запуска, healthcheck, cron/systemd примером.
• SQL-файл с используемыми запросами.
• Короткий runbook “что проверять, если посты перестали выходить”.

Сделай реализацию production-ready, с чистой архитектурой (config -> db -> repository -> service -> scheduler), и покажи финально:

• список env-переменных,
• SQL выборки/mark,
• как решена конкурентность,
• как проверено, что поведение совпадает с текущим.
