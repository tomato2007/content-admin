# Checklist `public.posts`

## Что должно совпадать между prod и dev

- `[x]` Таблица должна существовать как `public.posts`
  или быть эквивалентно настроена через
  `POSTS_SOURCE_SCHEMA=public`
  `POSTS_SOURCE_TABLE=posts`
  `POSTS_SOURCE_QUALIFIED_TABLE=public.posts`.

- `[x]` Колонка `id` должна существовать и быть стабильным PK.
  Она используется как `source_id` и как идентификатор picked record.

- `[x]` Колонка `content` должна существовать.
  Тип должен позволять хранить полный текст поста.
  По текущему контракту это основной источник текста публикации.

- `[x]` Колонка `media_url` должна существовать и быть nullable.
  Она нужна для разделения text/media сценариев.

- `[x]` Колонка `published_at` должна существовать и быть nullable `timestamptz`.
  По контракту запись считается “не опубликована”, если `published_at IS NULL`.

- `[x]` Семантика `published_at` должна совпадать.
  `NULL` = кандидат на публикацию.
  `NOT NULL` = уже опубликован, повторно не брать.

- `[x]` Типы колонок должны быть совместимыми с ожидаемой логикой.
  Минимум:
  `id` integer/bigint
  `content` text
  `media_url` text/varchar nullable
  `published_at` timestamptz nullable.

- `[x]` Данные в `content` должны вести себя как в проде.
  Если в проде есть мусорные хвосты, пустые строки, шаблонные вставки, это должно быть видно и в dev-срезе.
  Иначе очистка и фильтрация будут ложно “зелёными”.

- `[x]` Данные в `media_url` должны отражать реальность.
  Должны быть и записи с `NULL`, и записи с валидным media URL.
  Иначе нельзя проверить fallback `media -> text`.

- `[x]` Индексы должны быть эквивалентны или хотя бы совместимы.
  Минимум нужен индекс по `published_at`.
  Желательно проверить partial index для `published_at IS NULL`, если он есть или нужен в проде.

- `[x]` Поведение под конкурентной выборкой должно быть совместимо с Postgres.
  Таблица должна быть доступна для `SELECT ... FOR UPDATE SKIP LOCKED`.

- `[x]` Таймзона `published_at` должна быть согласована с продом.
  Если в проде это `timestamptz`, в dev не стоит заменять её на naive timestamp.

- `[x]` Объём данных в dev должен быть достаточным.
  Не 2-3 строки, а хотя бы десятки/сотни записей.
  Чтобы проверить случайную выборку, пустые выборки и отсутствие дублей.

## Что может не совпадать

- имя самой БД
- пользователь
- пароль
- хост
- порт
- SSL mode

## Что лучше проверить отдельно SQL-запросами

- структура таблицы `public.posts`
- индексы таблицы
- количество `published_at IS NULL`
- количество строк с пустым `content`
- количество строк с `media_url IS NOT NULL`
